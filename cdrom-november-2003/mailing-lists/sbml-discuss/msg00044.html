If you really wanted to, though I'm not sure if I support it or not at this
stage, one could separate SBML into two files, a file which indicates the
network structure (no dynamics) and another which refers to a network
structure model but adds all the dynamics information. You could even go
further and split the dynamics file in two, one which stores dynamical
information without values and a second which refers to the dynamics file
(which in turn refers to the structural file) but adds values. Whether this
is a good idea or not one would have to go through various scenarios to see
if it has a real advantage or not.

Herbert

I agree with the follwowing position:

> There seems to be a suggestion in the current discussion that parameters
> are "less objective" than model structure, and so can be "relegated"
> from the main model description. I hate to disillusion anyone, but I can
> assure you that your models suck just as much as your parameter values!
> ;-) Of course, that doesn't mean that the models aren't useful, but the
> same applies equally to parameters - even if you are uncertain about
> some parameter values, your "best guess" is a vital part of the model.
> You don't think twice about bunging down your "best guess" at model
> structure - why are you so worried about putting down your "best guess"
> at a parameter value?

I am for keeping parameter values together with the rest of the model. If
you have two models that only differ in the values of the parameters, why
not represent this with two SBML files? I do not see why there is a need
for separating parameter values from the chemistry and the kinetics.

I agree that we could extend the parameter tag to include a property that
indicates a parameter value is missing (unknown). This seems to work for
what Cliff wanted. I have to note that models that have unknown parameter
values are only useful for stoichiometric analyses, but this is a
legitimate use.  The property would help the rest of us who want to do
kinetic analyses, in which case these models need to be handled in a
special way (e.g. asking the user for "guess values" for these
parameters).

Finally, I would caution against extending SBML to cater for everybody´s
(all program´s) needs. If we do this we run the risk that SBML could allow
definitin of disjoint model structures, which means that some files could
be of no use to some programs and other files of no use to the other
programs. If this is to be an exchange format we must keep it withing what
is exchangeable!

Pedro Mendes
