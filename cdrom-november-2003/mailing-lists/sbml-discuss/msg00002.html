Hi Mike,

I didn't include quite enough information in my previous post.

The CellML language consists of a small number of syntactic constructs with 
simple relationships (even if the validity of a CellML document is governed by a 
complicated set of rules). A CellML document can thus be broken up and stored in 
a database in a straightforward manner. Developers have some freedom (i.e., 
problems) in how they deal with MathML and RDF.

Similarly, it should be possible to map most database representations of models 
into valid CellML 1.0 documents even if the process is lossy. The use of RDF 
ensures that all important metadata available in the database can be exported in 
a machine-understandable format.

My proposition remains that, as we move more biological information (including 
models) from files into databases, CellML will become more useful as an exchange 
format for moving models from databases to software and other databases, and 
less used as a storage format.
