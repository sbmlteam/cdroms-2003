The attached PDF contains the latest revised working draft of the
SBML Level 2 specification.   This will be the last revision
before the SBML forum in May.  We anticipate the community
adopting this as a final standard at that meeting.

The attached XML schema corresponds to the attached working draft.

The following are the changes we have made to the draft
that were requested at the last SBML forum in November 2002 in
Stockholm:

- parameter and initial values are all optional

- all elements are derived from SBase

The following are changes to the standard we made (each is justified,
feel free to comment):

- A functionDefinition element contains a MathML math
element instead of a Math lambda element.  The math element must contain
a lambda element.  This change is to ensure that SBML complies correctly
with the MathML standard

for example the following is a valid fragment:
    <functionDefinition id="f">
        <math xmlns="http://www.w3.org/1998/Math/MathML">
            <lambda>
                <bvar><ci> x </ci></bvar>
                <apply>
                    <times/>
                        <ci> x </ci>
                        <cn> 2 </cn>
                </apply>
            </lambda>
        </math>
    </functionDefinition>   

- the diff operator is not part of the valid set of MathML operators
	that can be used in SBML.  We are not aware of any
	simulators that capable of using this operator.  We are also
	unclear what the semantics of this operator would be in
practice.

- the URIs for the built-in delay function and time symbol
	have changed to http://www.sbml.org/sbml/symbols/delay
	and http://www.sbml.org/sbml/symbols/time respectively
	This form structures the sbml.org domain for future expansion.

- UnitDefinition element must contain at least one Unit element
	(previously they could contain zero Unit elements)

- Instances of AssignmentRule replace instances of
	ParameterRule, SpeciesConcentrationRule and
CompartmentVolumeRule.
	AssignmentRule is now a concrete structure with a 
	'variable' Sid attribute which refers to the variable to
	which the rule applies.  We think this is a useful
simplification.

  the following is an example of its use:
     <assignmentRule variable="s2">
         <notes>
             <xhtml:p>s2 = (k * t)/(1 + k2)</xhtml:p>
         </notes>
         <math xmlns="http://www.w3.org/1998/Math/MathML">
             <apply>
                 <divide/>
                 <apply>
                     <times/>
                     <ci> k </ci>
                     <ci> t </ci>
                 </apply>
                 <apply>
                     <plus/>
                     <cn> 1 </cn>
                     <ci> k2 </ci>
                 </apply>
             </apply>
         </math>
     </assignmentRule>

In the specification document we discovered a significant error in the
examples of event structures:
we used assignment elements instead of eventAssignment elements and in
used mathml elements incorrectly inside the assignment elements.

the following is a correct example of an event structure:

    <event>
        <trigger>
            <math:math>
                <math:apply>
                    <math:leq/>
                        <math:ci> P1 </math:ci>
                        <math:ci> tau </math:ci>
                </math:apply>
            </math:math>
        </trigger>
        <listOfEventAssignments>
            <eventAssignment variable="k2">
                <math:math>
                    <math:cn> 0 </math:cn>
                </math:math>
            </eventAssignment>
        </listOfEventAssignments>
    </event>
