> > Is your motivation of using XLINK to use generic XLINK 
> savvy programs 
> > to read SBML directly? If yes what are the benefits?
> 
> I want to store model definitions in databases.  I think XML 
> content in databases is migrating toward using the emerging 
> XQuery standard for reference and retrieval.  XQuery is 
> closely related to XPath 2.0. I expect XLink and XPointer 
> will evolve along a parallel course with the emerging XQuery 
> and XPath standards.  I think all of these capabilities will 
> become available in mainstream databases over time. Oracle 
> has much of this already (9iR2 has XQuery 1.0; not sure about
> XLink) and I expect Postgres and MySQL will have them soon if 
> they don't already.  If my component references are through 
> XLink, I expect the database will help me manage them (ie 
> dependency analysis through things like WebDAV).
> 
> Over time, software components will become available for 
> processing XPath, XLink, XPointer, ... much as there is now 
> with various parsers for XML, XSLT processors, and various 
> tools for schema design and management.  I hope there will be 
> enough capability in the generally available tools to relieve 
> some of the implementation burden for SBML processors.  We'll 
> only get those benefits if we use the community standards 
> wherever possible.  In some sense, this is an exercise to 
> determine if it can be done this way with a satisfactory end product.
> 
> I also think models, documents, and web content should 
> intersect. The more pieces I use from a common substrate, ie 
> W3C recommendations, the more likely these disparate views 
> will work together.  This is part of the rationale behind the 
> whole Model-As-Document view in the proposal.  In the work 
> we've been doing with our model building environment, users 
> pretty consistently want to be able to integrate model 
> definitions with various types of media: browsable content 
> (e.g. web sites), publication ready presentation material, 
> working documents, as well as what might be considered more 
> conventional uses like simulation or model analysis.  One 
> thing I've learned about content management over the years is 
> as soon as you have 2 sources for a piece of information, at 
> least one of them is out of date.  If I'm writing a paper and 
> I want to include some or all of the text of the model I'm 
> writing about, I need to be able to find the references to 
> the model components.  My document prep application will 
> (someday) understand XLink/decendent; it probably won't 
> understand a customized solution for SBML.
> 
> In short, composition can be thought of as a cross reference 
> between model definitions and XLink/XPointer/XPath is the 
> cross reference mechanism defined by W3C.  We've decided to 
> use an XML based representation for SBML so I think we should 
> use XML based technologies.
