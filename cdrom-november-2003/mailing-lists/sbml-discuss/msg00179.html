My understanding is SBML contains only structure and semantics of the model, and
does not specify what algorithm to be used to evolve the model.
Simulators can always check if such fragile variables are accessed in a improper
way according to their implemented algorithms.   SBML is a model exchange language.

Another thing is there must be a nomenclature system of Species and Reactions to 
make the submodels replaceable.    Then hiding such fragile Species, if needed,
can be very simple; for example put '__' before ID (Python approach), or, set 'hide' 
attribute to true.    In my opinion interface definition overkills here.


> > It is not necessary a case for E-Cell 3, since it uses asynchronous 
> > discrete event approach, but I understand the problem.  
> 
> Can you solve Index 2/3 DAE? 

No, currently, in a simultaneous fashion, unless it is provided as a plug-in Stepper.
Actually we are in process of writing plugins for it.   What we have so far are only
Gillespie-Gibson, Fehlberg2(3), Dormand-Prince 4(5)7M, ESSYNS (S-System and GMA),
and a Difference equation solver.   We will have a cellular automata module,  Tau-Leap,
and some implicit Runge-Kutta variants (the multi-algorithm framework works best with
one-step methods, so we won't have PC or BDF types of solvers).
