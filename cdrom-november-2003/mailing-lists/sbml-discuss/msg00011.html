I'm afraid I can't resist joining the argument regarding models and
parameters...

I'm a bit worried about the "downgrading" of the status of parameters. One of
the things I like about SBML (as it stands) is that it encapsulates everything
one needs in order to simulate from a model, and that it says nothing about
the simulation strategy (deterministic, stochastic, etc.). 

There seems to be a suggestion in the current discussion that parameters are
"less objective" than model structure, and so can be "relegated" from the main
model description. I hate to disillusion anyone, but I can assure you that
your models suck just as much as your parameter values! ;-) Of course, that
doesn't mean that the models aren't useful, but the same applies equally to
parameters - even if you are uncertain about some parameter values, your "best
guess" is a vital part of the model. You don't think twice about bunging down
your "best guess" at model structure - why are you so worried about putting
down your "best guess" at a parameter value? 

Of course, as someone has already mentioned, parameter values are often
uncertain, but that doesn't mean one shouldn't try to encapsulate the current
state of knowledge regarding that parameter in the model. In one of my current
research projects, I use a cut-down version of SBML level 1, with some slight
tweaks. One tweak I use is for the parameter tag. As well as having a "value"
attribute (which is compulsory!!!), there is also an optional "sd" attribute,
representing a standard deviation to be associated with the mean "value" (or
"best guess"). If the sd attr is missing, the parameter is assumed to be
known. If it is present, there is assumed to be a gamma probability
distribution associated with the parameter, whose mean and standard deviation
are given by "value" and "sd" (there are deep theoretical reasons for adopting
gamma distributions for parameters in stochastic kinetic models). 

This "sd" attr has a number of advantages. Simulation software can ignore the
sd attr, and carry out a simulation on the basis of the "best guess". On the
other hand, if one is interested in parameter sensitivity, one can sample a
representitive set of parameters from the probability distributions associated
with the parameter values, and then run multiple simulations based on
different parameter samples. Even if one requires something crude such as a
"range" of parameter values, this can be constructed either as a mean
plus/minus x standard deviations, or (better), as a y% equitailed gamma
probability interval. Finally, if one is interested in making inferences for
the parameters on the basis of time-course experimental data, one can use the
distributions to represent "prior knowledge" regarding the parameters before
observing the experimental data, which can then be refined to give improved
parameter estimates using a suitable statistical inference framework.

People have said that they would like parameter sets in separate files, so
that they can run alternate sets with the same model. I would suggest that
there is just as much uncertainty about models, so what people really want is
a collection of alternate sub-models in separate files (complete with
associated parameter specifications!). 


Jonathan Webb wrote:
>  - Model definitions exist without reference to specific _numeric_
>    values for at least some of their parameters.

But I would argue that this is barely better than a "schema" that a
non-numerate biologist would produce. Models have to be quantified to be
predictive, and they have to be predictive to be really useful.

Just my two pence worth...   ;-)
