> Hello all,
> 
> we would like to present another suggestion for the sbml 
> layout extension.
> 
> For that we took the basic idea that information about the 
> position and 
> relations between the graphical objects should be separated 
> from render 
> information like  line thicknesses, colors, shapes. We only 
> deal here with 
> the first. 
> We took from the discussion in this forum what we think are 
> the important 
> requirements (that our first suggestion did not meet).
> 
> The requirements we considered are:
> 
> 1. several layouts for one model in one file.
> 
> 2. layout separate from the model in the file.
> 
> 3. one model object (compartment, species, reaction, species 
> reference) can be 
> represented by several graphical objects.
> 
> 4. graphical objects that do not refer to model objects are 
> possible (Of 
> course graphical objects can refer to a part of the model and 
> usually do).
> 
> We provide a sample sbml file and a png file with the 
> rendered layout. These 
> files were generated manually and the sbml code is based on a 
> hypothetical 
> sbml level 2 that would have id´s for species references.
> 
> 
> Some additional notes:
> 
> -the basic structure of this suggestion is similar to John 
> Wagner´s draft 
> (that reflects requirements 1 and 2), while our first try was 
> similar to 
> Herbert Sauro´s suggestion.
> 
> - the ...GR in the tag names stands for graphical 
> representation (but the 
> actual names of the tags don´ t really matter that muchin 
> this phase of the 
> discussion)
> 
> -all the graphical objects have an id. This id is used only 
> in the case of the 
> speciesGR (it is needed in the reactionGR). But we think that 
> every object in 
> sbml should have an id.
> 
> - the "ref" attribute in the graphical objects is optional 
> and refers to the 
> id of a model object.
> 
> - there are two kinds of graphical objects that don´t refer 
> to a model object: 
> first the ones that _could_ refer to a model object but don´t 
> (since the 
> "ref" attribute is optional). This means you can have species 
> and reaction, 
> ... in the layout that are not in the model; second the 
> objects that are in 
> the "listOfAdditionalGraphicalObjects". In our example we 
> have put only a 
> text label object in this list, other types of objects would 
> be possible.
> 
> -  since a species reference in the model connects the 
> reaction with the 
> species, and the arrows (or arcs) in the diagram  connect the 
> reactionGR with 
> the speciesGR, we think the arcs (or arrows) should refer to 
> the species 
> references. To reflect that we have called them speciesReferenceGR
> 
> - the possibility to have several speciesReferenceGR objects 
> that refer to one 
> species reference in the model solves the following problem that was 
> mentioned in the discussion on the list: The user might want 
> to render the 
> reaction A->B with encyme E as a reaction A+E->B+E. So we 
> have 3 species 
> references in the model (one substrate, product, modifier 
> each) but 4 arcs 
> (two substrate, two product). So two speciesReferenceGRs (one 
> substrate, one 
> product) refer to one species reference (modifier) in the 
> model. This is the 
> reason why we put the "role" attribute in the 
> speciesReferenceGR tag. It 
> should be optional. If it is not there the role is taken from 
> the species 
> reference. 
> 
> -Right now the speciesReferenceGR specifies only startpoint 
> and endpoint of 
> the arc. It might make sense to include coordinates for an 
> arbitrary number 
> of points in the speciesReferenceGRs to describe the  the arc. 
> 
> We hope that our suggestion can be the base of some 
> discussion. Did we miss 
> anything important? Please comment.
>
