Here's a reply to your questions:

> 1. The E coli flux balance model associates one or more gene symbols,
> enzyme names, and EC numbers with each reaction.  Right now the gene
> symbols and enzyme name information is only in the annotations at the
> beginning of the model, and since the suggestion to include 
> the EC numbers
> was not made until after I made this sample, those are not 
> even in notes;
> is there a way of incorporating the gene symbols, enzyme names, and EC
> numbers into the model proper?  There are cases where more 
> than one gene
> is associated with a single reaction as well as cases where a 
> single gene
> is associated with several different reactions. 
> 

SBML Level 2 uses the CellML metadata specification to define how
to attach this kind of data to a model.

http://www.cellml.org/public/metadata/cellml_metadata_specification.html

In particular please look at the section

http://www.cellml.org/public/metadata/general.html#sec_bio_entity

This section shows how to associate a resource with, for example,
a SWISS-PROT identifier.

An example is given:
http://www.cellml.org/public/metadata/general.html#fig_bio_entity_exampl
e

That example could be incorporated into SBML Level 2 in the following
way:

<?xml version="1.0" encoding="UTF-8"?>
<sbml xmlns="http://www.sbml.org/sbml/level2"
		level="2" version="1">
  <model name="simple">
    <listOfCompartments>
      <compartment id="c1" />
    </listOfCompartments>
    <listOfSpecies>
      <species metaid="id0" id="X0" compartment="c1"
 				   boundaryCondition="true"
                        initialAmount="1">
		<rdf:RDF
	
xmlns:cmeta="http://www.cellml.org/metadata/1.0#"
	
xmlns:dc="http://purl.org/dc/elements/1.0/"
	
xmlns:dcterms="http://purl.org/dc/qualifiers/1.0/"
	
xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#
	
xmlns:rdfs="http://www.w3.org/2000/01/rdf-schema#>
			<rdf:Description rdf:about="#id0">
				<cmeta:bio_entity>
					REST OF CELLML EXAMPLE GOES HERE
				</cmeta:bio_entity>
			</rdf:Description>
		</rdf:RDF>
      </species>
    </listOfSpecies>
REST OF MODEL GOES HERE

I'd be interested if you think we should extend the set of identifier
types
defined in the CellML specification.

> 2.  The initialAmount value is not among those designated as Optional
> in the species data type, so I have specified it as 0 in all cases.
> The truth, however, is that initial Amounts are not in any 
> way associated
> with the flux balance model, and it would be more appropriate not to
> specify any value.  Is it OK not to specify initialAmount values that
> make no sense?
>

SBML is designed to support a variety of network analyses and should
continue be extended to support more types of analysis.  It
is inevitable that there will data which essential for one type of
analysis and not for another.

An example of this is having kinetic laws optional: some connectivity
analysis can be performed without these laws.

Thus there is a strong argument that initialAmount should be also
optional.
This would be useful in that software can then decide if the model is
appropriate to it.  At the moment software can't tell if the species
amount is valid or not.
 
In the meantime I think it is acceptable to have incorrect values for
initial amounts although I would notify recipients that the model
doesn't
support certain analyses including conventional time based simulation.

> 3. Many of the fluxes have lower and upper bounds for their 
> values (which
> can be both positive and negative).  These are not kinetic 
> parameters but
> are related to rates.  Although these can be adjusted to 
> study the effects
> of mutations, the default values are included in the native form flux
> list. How do we represent these in SBML 2.0? 
> 

There is no current mechanism for representing this in SBML Level 2.
Perhaps we should consider this for inclusion in this or future levels
of SBML.

In the meantime you should use a the annotation mechanism that is
described 
in the SBML Level 2 document for this data.

I hope this helps.  If you have any further questions please don't
hesitate to contact me.

Aside: In general I agree with Cliff Shafer's comments that variant
initial conditions and
parameters need to be stored separately from a model.  I think the model
should retain default values, particularly for publishing the best
estimates
and/or measurements for wild type organisms and to enable analyses to
operate
on a single file, for example download from the web, by default.
If we adopt a simple reference mechanism for parameters and initial
conditions
we can then have simple files that consist of sets of reference and
alternative
value pairs.
