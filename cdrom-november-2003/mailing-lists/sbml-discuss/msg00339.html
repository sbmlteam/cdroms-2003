The whole reason for including a resolution was so that
we wouldn't need to specify units--all units would be in
"points". Also, I didn't say we should use 1/72 as the
standard unit, I said it should be specified and, in the
event it wasn't, then the default should be 1/72, but really,
I prefer to simply specify that the units are in points (72
DPI), and then not allow it to be specified. Originally,
I thought, why not let people read/write in their own
units (like Herbert's 96 DPI), but then I realized, that
would only be of benefit when writing; when reading,
you'd still be forced to convert to your units from the
units specified in the SBML file.
 
So yeah, I agree--units should be set in stone as "points"
and not allowed to be overridden; this also gets us out
of the political argument over metric v. English.
 
I also agree that we shouldn't throw things in just because
they are convenient; in this case, it's not a matter just of
convenience, it's a need for certain types of tools (the
printer tool was a good example), and a convenience
for others.
 
As for Eric's suggestion about compound role, I think
we're going to have to really think about this, and hard,
the principal problem being that currently, the Species
and SpeciesReference elements have no way of specifying
such info, so we can't just say it belongs there unless
everyone is willing to not have such info until the Species/
SpeciesReference are changed. On the other hand, if that
is where we decide such things go, then we will expand
the scope of the DWG group to include "fixing" those.
But as I said, we need to really think hard about this.
How do we start? Maybe Eric could put together a
fairly detailed description of requirements/wants?
 
John

	OK, now I guess I understand. The problem is, we tried to limit the
	layout information to the parts that are "absolutely" necessary which
	obviously did not include units as was pointed out by Herbert Sauro and
	John Wagner. I am not really sure, if we really have all the necessary
	bits and I hope that we will continue to get feedback about it.
	Now some people start to miss bits that can be specified as
	"convenient". So we have to decide wether this information should
	really be put into the layout part, or if it would not better fit
	somewhere else like in the render information (to be specified) or the
	model itself.
	I guess if units were to be put somewhere it would rather be in the
	layout part then in the render information, so if everybody could agree
	that having optional units specified for the layout information, we
	will put it there.
	I also had a look at the SVG specification and they seem to use pixel
	as the standard unit if no other unit is specified. All measures in SVG
	can be given in different units. In my opinion this would be an
	overkill for SBML and I think it would be easiest to agree on the
	suggestion made by John Wagner to use 1/72 inch as the standard unit.
	So what do others think? Should units be included or should we just
	agree that all values are 1/72 of an inch and if programs don't like
	that, they can do whatever they want? And if we want unit support, do
	people really need to be able to specify units in pt, mm, inch etc.?
	As to the suggestion mad by Eric Minch on including information about
	the role of certain compounds, I personally think this information does
	not belong into the layout part. I think this should rather be put into
	the model if necessary (e.g. a new attribute in reaction that specifies
	wether this is a transport or an attribute in SpeciesReference that
	determines the role of the modifier (if it is used as one).
	If this information is just used for rendering purposes, it should go
	with the render information.
	Any thoughts on this?
	
	Ralph Gauges
