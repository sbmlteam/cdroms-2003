In the case of scripting tools, such as Jarnac or SCAMP, it is up to the
user to realise the consequences of naming parameters. These tools are for
experienced users and I thus expect them to understand how a scripting tool
works. eg the two Kms in the following script are assumed to be identical:

S1 -> S2; Vm1*S1/(Km + S1);
S2 -> S3; Vm2*S2/(Km + S2);

In the case of Jdesigner, things happen differently. If the user uses the
predefined rate laws (which is probably most of the time) then they are
completely insulated from setting parameter names. In the SBML, all
parameters in rate laws are prefixed with the name of the reaction, eg J1_Km
etc. It's still a global space because that's how the internal engine in
Jarnac works. If a user wants the same parameter in different rate laws,
then they use the free format option which allows them to do anything. In
this case they need to know a bit more about what's happenning. But I've not
heard anyone complain yet.

Herbert

Just to get some feedback...those of you who do *not* have
(separate) namespaces, how do you mangle/convert the local
parameters in KineticLaw? I guess one obvious approach is
to blindly throw them into the global namespace and ignore
the consequences, and another is to check to see if throwing
them in the global namespace would result in a collision
and tell the user "I'm sorry, Dave, I'm afraid I can't do that".
A "better" way might be to mangle the name, converting
Reaction R's kinetic parameter V to, say, R_V (appending
a number or something until it is unique with respect to
the global namespace), but then you'd have to substitute
the new name back into the KineticLaw's expression
(not to mention that the user might be a little distressed
to suddenly see his/her parameter name changed).
 
I'm not asking because I want to change the spec, I'm
asking because I am trying to implement this, and it's
still a problem. Currently, I'm throwing them blindly
into the global namespace (ugh!).
 
John
