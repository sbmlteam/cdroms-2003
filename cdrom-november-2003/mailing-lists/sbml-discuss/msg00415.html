I am still trying to understand what different people expect from the sbml 
layout definition.

In our suggestion we have graphical representations of the different  objects 
in the model. In our case this was one representation for each reaction, 
speciesreference, and compartment; and possibly several representations for 
every species. 
John said in a recent mail that he wanted several possible representations for 
all objects in the model. That makes sense.
So far this would still maintain that every graphical object refers to exactly 
one object in the model.
(if we allow multiple representations of species references this also solves 
the problem that we might want to visualize A->B with catalyst M as A+M->B+M; 
we still think that an arc in a reaction is the graphical representation of a 
speciesreference)


I guess some people would like to be more flexible and allow graphical objects 
that do not refer to a specific object in the model. 

The overall difference would be that in the first case we would define a 
layout *for* a specific model. In the second case we would define a rather 
general layout parts of which may have a relation to the model.
(in the implementation that means that the field in the graphical objects that 
contains the id of a model object ist mandatory in the first case and 
optional in the second)

So does someone have the need to store layout objects that do not 
correspondent to any object in the model?

Sven

ps: I think we really should  give the species references an ID as soon as 
possible
