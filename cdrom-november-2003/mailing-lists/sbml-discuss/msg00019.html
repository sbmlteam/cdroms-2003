With respect to setting parameters and initial conditions, why not simply
let these things have two states, defined or undefined. This is what I do in
my simulation packages because if one doesn't it's very easy to forget to
initialize something, by default these uninitialized values would be zero
and this introduces subtle 'bugs' in to the model. So unless a variable or
parameter is actually set, it remains undefined and the simulator can flag
the user than his/her model has undefined values. In SBML we could have some
way of indicating whether a parameter, initial condition has a defined
value. For a flux analysis, the initial conditions could be set to undefined
(I presume there aren't any rate laws). This would allow a user to turn this
flux model in to a full dynamic model at a later date if they wished. I
believe that values for parameter values are part of the model (I guess it
depends on your definition of a model and the type of model you're
considering) and I think forcing users to separate them out into a separate
file would not be good. At least make it optional. Note that the value of a
rate constant represents our model of the reaction characteristics of the
process, hence I would consider it to be part of the model (unless it's a
flux model when it doesn't apply).

As regards different uses for SBML perhaps there should be some sort of tag
at the beginning of the model to indicate the models' intention, eg flux
analysis, structural analysis, dynamics simulation, mutation sets,
stochastic, etc. I've already run into this problem with respect to
stochastic and deterministic models since I treat stochastic models slightly
differently in that the rate laws in a stochastic simulation are derived by
the tool rather than the user (other wise users simply enter invalid rate
laws as I saw in a recently accepted paper!) and at the moment I've no way
of knowing what type of model is stored in the SBML and by the sounds of it
people one everything including the kitchen sink to be represented in
SBML/MDL ! :)
