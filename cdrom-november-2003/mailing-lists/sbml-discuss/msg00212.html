Andrew Finney wrote:
> We propose in Level 2 to
> introduce
> two concrete subclasses of AssignmentRule: ScalarRule and RateRule.  

Actually, only the ScalarRule is really an Assignment, since it changes 
the value by replacing the old value.

The RateRule specifies the rate of change and is well named in this 
respect. There is not much more in common between (using Level 2 Draft 3 
language):
      <AssignmentRule type="scalar"> and <AssigmentRule type="rate">
than between
	<AssignmentRule type="scalar"> and <AlgebraicRule>

The only thing in common is the Sid field, bewteen the two 
AssignmentRules, but conceptually there is more difference than commonality.

Therefore I propose option c) to make the hierarchy flat:

There are 3 kinds of rules, all derived from <Rule>:
<AlgebraicRule variable: Sid>
<AssigmentRule variable: Sid>
<RateRule>

All three have different default units:
<AlgebraicRule>	: no default units
<AssignmentRule> : same units as the units of the variable referred to 
in the Sid field.
<RateRule> : same units as the units of the variable referred to in the 
Sid field, divided by the units of "time".


The advantage is that existing models that do not use <AssigmentRule 
type="rate"> do not change, since <AssigmentRule> already has the 
default type="scalar", and therefore the same meaning as the 
<AssignmentRule> in my proposed solution. In Andrews proposed solution 
b), AssignmentRule is abstract and cannot be used.
