>
> SBML is designed to support a variety of network analyses and should
> continue be extended to support more types of analysis.  It
> is inevitable that there will data which essential for one type of
> analysis and not for another.
>
> An example of this is having kinetic laws optional: some connectivity
> analysis can be performed without these laws.
>
> Thus there is a strong argument that initialAmount should be also
> optional.
> This would be useful in that software can then decide if the model is
> appropriate to it.  At the moment software can't tell if the species
> amount is valid or not.
>
> In the meantime I think it is acceptable to have incorrect values for
> initial amounts although I would notify recipients that the model
> doesn't
> support certain analyses including conventional time based simulation.
While the current specification for SBML forces one to include 
initialAmount values, in support of Andrew I would strongly argue for 
this to be made optional. Firstly, the initialAmount is a 
simulation-specific, not a model-specific, attribute. Ideally, such 
simulation-specific information should be kept in a separate file to 
provide a clear separation of model (topology and mathematics) from 
data (initial/boundary conditions) and combined with the model at 
simulation time. Secondly, it is dangerous practice to include 
information in a model that is not well supported by experiment, 
especially if such information has been 'plucked out of the air'. While 
your simulation may not depend on such values, other simulations using 
the SBML representation of your model might well make use of the 
values, unaware of their dubious heritage, yielding incorrect 
simulation results. Should we consider this more closely at Stockholm?
>
>> 3. Many of the fluxes have lower and upper bounds for their
>> values (which
>> can be both positive and negative).  These are not kinetic
>> parameters but
>> are related to rates.  Although these can be adjusted to
>> study the effects
>> of mutations, the default values are included in the native form flux
>> list. How do we represent these in SBML 2.0?
>>
>
> There is no current mechanism for representing this in SBML Level 2.
> Perhaps we should consider this for inclusion in this or future levels
> of SBML.
...or allow a more general use of MathML so that such constraints can 
be included explicitly. MathML's 'piecewise' function is able to 
represent such mappings.
>
> In the meantime you should use a the annotation mechanism that is
> described
> in the SBML Level 2 document for this data.
