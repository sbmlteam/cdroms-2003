> I haven't decide my stance on this issue, but at least
> I agree it can be useful to have rational and fractional 
> stoichiometries because I got some requests to support them 
> from users.
> 
> One thing I can say is it is also useful to have an attribute 
> to indicate the type of the stoichiometry:  such as one of 
> 'integer', 'rational', and 'fractional'.
> 
> The information can be used in two ways:
> 
> (1) Checking:  if the simulator of the simulation algorithm 
> supports it.
>                        For example, Gillespie usually 
> supports only elementary reaction schemes,
>                        which means that the stoichiometries 
> are usually integers 1 or 2 (or 3..).
>                        In scientific programming float( 1.0 ) 
> must be distinguished from int( 1 ).   
>                        And frac{1.0}{1}  != 1, although 
> frac{1}{1} can be equal to int( 1 ).
> 
> (2) (less importantly) Optimizing: 
>                         If you have ever written simulation 
> code, you know what I mean.
>                         For example, in Gillespie code I 
> could have been able to eliminate several
>                         conditional statements (which can 
> stall super-pipiline in modern RISC CPU)
>                         by assuming stoichiometry is integer.  
> 
> 
> By the way, E-Cell currently does support  only integer stoichiometry.
> 
> -Kouichi
> 
> 
> > SBML Levels 1 and 2 so far have stipulated that the 'stoichiometry' 
> > field on species references in reactions must have a 
> positive integer 
> > value.  To achieve fractional stoichiometries, SBML models are 
> > supposed to use the additional 'denominator' field (also a positive 
> > integer) to create a fraction.
> > 
> > Some users have pointed out that this is not the same thing as 
> > actually allowing stoichiometries to be floating-point numbers.  In 
> > certain modeling situations (such as flux balance models or other 
> > phenomenological models), a user may want to record stoichiometries 
> > that are not integers.  For example, Wayne Rindone's models contain 
> > lumped reactions with floating-point stoichiometries whose 
> values were
> > derived experimentally.  While these numbers could be
> > converted into fractions ('stoichiometry'/'denominator') by
> > the underlying software, this may not have the same meaning
> > in the modeling context as the (originally measured)
> > floating-point stoichiometry value.
> > 
> > A second issue is the question of whether to allow stoichiometry 
> > values to be expressions rather than only numbers.  A 
> simple example 
> > of the utility of doing this arises in models using non-integer 
> > stoichiometries: the stoichiometry value may be the same in 
> two of the
> > 'speciesReference' structures in a reaction, and it would be
> > useful to be able to define a parameter for the value and
> > using that parameter instead of repeating the same number
> > (because the latter is more error-prone).
> > 
> > Finally, the logical conclusion of the ideas above would be 
> to get rid 
> > of the 'denominator' field entirely, and allow 
> 'stoichiometry' to be a 
> > MathML expression.  Then fractions could still be expressed 
> (using the 
> > MathML equivalent instead of 'stoichiometry'/'denominator'); in 
> > addition, parameters could be used naturally, and it would 
> leave room
> > to grow in the future for more exotic formulations, should
> > the need ever arise.
> > 
> > Several options come out of considering these different
> > issues:
> > 
> > For the 'stoichiometry' field:
> > 
> > a) don't do anything; keep the 'stoichiometry' attribute as
> >    it is currently defined (a positive integer)
> > 
> > b) change the 'stoichiometry' attribute to have type
> >    'double'
> > 
> > c) change the 'stoichiometry' attribute to allow values of
> >    type 'double' as well as constant parameter id's
> > 
> > d) change the 'stoichiometry' attribute to be a MathML
> >    expression
> > 
> > For the 'denominator' field:
> > 
> > i) Keep the 'denominator' as-is (a positive integer)
> > 
> > ii) Remove 'denominator'
> > 
> > Please reflect on and discuss these issues on the list.  The voting 
> > system will be enabled beginning this Friday and we will 
> post voting 
> > information at that time.
> > 
> > If you have questions about this topic or the reason for the vote, 
> > please feel free to call Mike or Andrew directly.
> > 
> > -- Mike & Andrew
