FYI

the SBML Level 2 document states in section 4.1:

"The character encoding for SBML is UTF-8. SBML documents should include
the encoding
attribute with the value UTF-8 in the XML prologue."

SBML Level 1 doesn't have this text as far as I remember.

I don't know if that helps.

My vote would be to specify that SBML is always encoded in UTF-8
and follow through with that in the MIME registration.
Would that make SBML unusual for an XML format?
Do MIME processors need the flexibility to transform UTF-8 into
us-ASCII?
i.e. what are the concrete implications? if we do this are we imposing
an unrealistic
restriction on transmission that would probably not happen in practice?

Can we make the char set non-optional and restricted to UFT-8?
If yes this would then avoid the issue with rewriting to disk.
If we restrict the car set does that make interpreting the incoming
stream 
because the its in only char set?

>  
> Chris Lilley on the ietf-types@iana.org mailing list wrote: 
>  
>    Do the existing SBML-consuming tools honor the charset 
>    parameter when SBML is sent over HTTP or email?  And where 
>    the charset parameter disagrees with the encoding in the XMl 
>    encoding declaration, do these tools rewrite the XML when 
>    saving it to disk? 
>  
>    Note that, although charset is optional in this registration, 
>    RFC 3023 still imposes requirements on software in the 
>    *absence* of a charset. 
>  
>    If not, I suggest removing this optional parameter as 
>    follows: 
>  
>         There is no charset parameter. Character handling has 
>         identical semantics to the case where the charset 
>         parameter of the "application/xml" media type is 
>         omitted, as described in [RFC3023]. 
>  
>  
> I'd like to hear your opinions on this. 
>  
> The default handling referred to above is to assume that the 
> application/sbml+xml entity is encoded in us-ascii when it's 
> sent over HTTP. 
>  
> If we follow Chris Lilley's suggestion, then we would *not* be 
> able to specify that an application/sbml+xml entity sent over 
> the web or through email is encoded in, say, utf-8 or 
> iso-2022-jp. 
>  
> If we *do* follow his suggestion, then we would be imposing a 
> requirement on all software that would directly read SBML models 
> encoded as MIME entities: we would be requiring that they deal 
> properly with encodings other than us-ascii. 
>  
> To get a handle on the relevant issues, you might want to read 
> the relevant section of RFC 3023.  It's at: 
>  
   http://www.rfc-editor.org/rfc/rfc3023.txt 
   section 3.1 (the full discussion takes 1 page) 
 
If I don't hear from anyone, I'll incorporate Chris Lilley's 
suggestion and disallow the charset parameter.
