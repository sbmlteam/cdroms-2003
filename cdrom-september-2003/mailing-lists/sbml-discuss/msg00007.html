I think the concensus at the moment is that:

a) we should have optional parameter values

b) they should not default to anything
	its upto tools how they handle this - ideally they 
	should complain

c) we should have parameter value sets which can be incomplete

d) parameter sets refer to models

I think (c) implies that parameter sets can be stacked up

Its not clear if parameter sets should be internal/external to the model
or how parameter sets will work with modularity

> I am stepping into this and know I will be kicked :)
> 
> Default values are not a part of the model. Default values 
> are part of an user interface, e.g., a model editing tool. 
> 
> The model itself has values for each needed parameter. Which 
> parameters are mandatory or optionally I will not discus 
> here. By introducing default values as part of the SBML 
> language you complicate the parsing of an SBML file. This is 
> something I would very much like to avoid. 
> XML is not normalized and therefore it will repeat 
> information. We should not try to normalize SBML by 
> introducing default values.
> 
> SBML is intended to be an exchange standard between tools. If 
> default values are introduced all tools would have to 
> implement them to understand the SBML file at hand. This 
> complicates the tool development and will likely keep the use 
> of SBML to a minimum. We should avoid everything which slows 
> the spreading of SBML.
