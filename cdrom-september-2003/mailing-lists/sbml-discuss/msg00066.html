Hi Andrew and Jonathan,

First of all, thanks for the proposal and for starting this thread.

On 05-Mar-2003 Andrew Finney wrote:
> a) XPATH can only refer to concrete XML structures.  Structures that are
> inferred
> by SBML - in the future - such as the inside of a specific model
> instance can't be referred
> to (if you allow any number of instances of the same sub-model).  I
> guess it can be done by
> a pair of XPATHs one to the instance and the other to inside the
> sub-model.

Is this a thing, a XML-XPATH-Library will automatically understand? 
Does it help us?

My idea of referencing was different but similar:

- what we really need is references to *instance* structures, not to
  the original class definition files.
- one should expect, that these instances and the referred elements are IDed
- My reference proposal would have consisted of an attribute of a type 
  similar to IDREFS that is interpreted as a path of instance (submodel) 
  and element (reaction, specie, parameter) IDs:
        
        "A B C" meaning e.g component C in submodel B of submodel A
        
- I wanted to use the same path for identifying elements in 
  parametrization of modelInstances
- Of course the identification of elements and checking of reference consistency
  has to be done manually

My main questions is:
Of course we can stop inventing our own path structure, if xpath does the 
job automagically. But with the comments of Andrew and my own readings I doubt 
this.
I think (as far as I understand XPATH) with Andrews proposal of two paths, one 
to find the instance and the other to find the element in the instance we 
have the recursion problem: What happens for changing the initial value of 
the specie Glc in the modelInstance A of type TA *inside* the modelInstance B of type TB 
of our current model. IMHO we need 3 paths (and so forth) there? 


> I have an open mind on this mainly to avoid reinventing the wheel.

Me too.

I have no problems, using any path representation, since I already have my
own (slot-)paths in my tool, that just wait to be translated ;-).
But on the other hand I wanted to keep things simple, and to use more complex 
technology only if there is an real advantage.
