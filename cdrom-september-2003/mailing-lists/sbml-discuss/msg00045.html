        In the case of large models (e.g. our whole cell E coli model) 
there is a real value in separating (at least some) parameters from the 
model.  When you have files in the multi megabytes duplication of common 
parts is to be avoided.
        e.g. If you want to run simulations where you are going to explore 
the differences in one or two substrates on the growth of a cell, but want 
to run lots of simulations, it makes good sense to have the separation 
between parameters (changed many times and a few Kb each simulation) and 
the model (one copy - several Mb).
        This is, of course, not necessarily how the DBsolve project plan 
to use SBML. Nor does it necessarily reflect the views of the DBsolve 
project team.

I am for keeping parameter values together with the rest of the model. If
you have two models that only differ in the values of the parameters, why
not represent this with two SBML files? I do not see why there is a need
for separating parameter values from the chemistry and the kinetics.

I agree that we could extend the parameter tag to include a property that
indicates a parameter value is missing (unknown). This seems to work for
what Cliff wanted. I have to note that models that have unknown parameter
values are only useful for stoichiometric analyses, but this is a
legitimate use.  The property would help the rest of us who want to do
kinetic analyses, in which case these models need to be handled in a
special way (e.g. asking the user for "guess values" for these
parameters).

Finally, I would caution against extending SBML to cater for everybody´s
(all program´s) needs. If we do this we run the risk that SBML could allow
definitin of disjoint model structures, which means that some files could
be of no use to some programs and other files of no use to the other
programs. If this is to be an exchange format we must keep it withing what
is exchangeable!
