Hello,

Herbert, Rowan, and Akira suggested to have a line-type field and some 
additional information that needs to be interpreted according to what line 
type was chosen (polylines, splines, bezier, ...). 
This is one of the possibilities that we were also considering. For discussion 
I want to contrast this with the other possibility (while I´m not shure 
myself what I would prefer):
We could also focus on one way to describe curves that can describe all curves 
that we would like to describe (although with unecessary overhead for 
straigth lines). This could be a sequence of curves (I mean the curves that 
are descibed by start point and end point and two additional points that 
define the tangents to these points, I believe they are called bezier, but 
I´m not shure about the terminology). This seems to be the way drawing 
programs like adobe illustrator (or kontour) work.

The first way (different curve types) would be easy to write, since all 
programs could write just like they internally represent the curves, but 
might involve conversion calculations while reading (if the curves are given 
as a curve type that my program doesn´t use internally).
The other way (just one general curve type) would be easy to read (at least 
only one conversion routine is needed) while it might be difficult to 
translate the internally used curves to the one standard curve type in the 
sbml output.

Sven
