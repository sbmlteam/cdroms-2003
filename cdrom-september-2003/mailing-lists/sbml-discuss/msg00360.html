> As I mentioned in the previous email I wasn't entirely sure 
> what was the practical side of this.
Exactly. Me either. It looks like all it does is separate it
from the SBML file. You still have to agree upon what comes
out of the application of the stylesheet. I guess one practical
effect would be that we could make changes to the display
spec without having to increment the version counter on the
SBML spec (which is actually a pretty good argument, I guess).

> Stylesheets sound like a nice idea, I'd be interested in
> hearing more about this but in more concrete terms. I'm not
> good at thinking very abstractly which CS people are so used to. 
Try this: http://www.w3.org/Style/XSL/
There are articles and how-to docs you can peruse. BTW, IMHO,
there are no simple examples, because there is no such thing
as a simple XSL example, and this is one of my primary concerns
with doing display via XSL. 

> The only issue I can think of is that people would have to 
> make sure they copy two files instead of one when passing
> models about, this might be a bit of a hindrance.
Unless the SBML files coming out of JDesigner (as an example)
always used the same stylesheet, in which case it could be
made available via the web (like we do with the SBML DTD).
However, generating an XSL file for each XML file is not
generally how things are done, and I don't think that's what
we have in mind here. XSL would be a good approach if we
always wanted models to be displayed in one way; otherwise,
we'd have to generate an XSL file every time we generated an
SBML file, and we'd be stuck copying two files rather than one.

> One other possiblity is to write translators which translate 
> one format, eg cellDesigner (formely SBEdit) to VCell to
> JDesginer at least as an intermim measure. The stylesheet
> project sounds quite ambitious.
Yes, but so does the format translator project. :)
