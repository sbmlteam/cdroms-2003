Andrew -- Good to hear from you! Unfortunately, I will be unable to attend
the SBML meeting. Here are a few comments.

I'll mention that we are finally getting up to speed with true use of SBML
Level 2.0. Marc Vass is completing his translator software from the
JigCell Model Builder to SBML Level 2.0. This means we are getting true
hands-on experience with converting our models to SBML 2.0. So its not a
theoretical discussion for us so much as it used to be. Note that by SBML
2.0, we mean support for what is in that appendix... we are assuming that
this will be included.

Beyond that, this issue of initial conditions and parameter values has
certainly come up before. It will need to be addressed in some way. It
becomes increasingly clear to us that the IC/Param values need to be
separated from the rest of the model, since these vary on a more rapid
development schedule. That is, a modeler is likely to set his equations
and leave them, while tweaking parameter sets frequently. Thus, the
"model" part needs to be stable while the "values" part changes
frequently. Ideally these would be in separate files. The notion of
include files would help a great deal here. It might be premature to deal
with that. But as a stopgap, making paramter/IC values optional is good
step. From our point of view, the idea that parameter values are a
required part of the model is totally wrong. What happens in practice is
that default value of (for example) zero gets stuck in... with the
horrible result currently that one cannot distinguish a default
placeholder (put there only because SBML requires it) from a value that
has been deliberately set to zero.

It would be helpful to us if the named functions had some sort of optional
tag that let us distinguish rate laws from other types of functions. The
reason is that we wish to display to the user of our model builder
software a list of available ratelaws, without including other functions
that are not meant to be used as ratelaws.

Finally, we do have need of variable parameter lists. The canonical
example is the mass action law, with variable numbers of parameters to be
multiplied together. We can live without this for now (we have Mass Action
1, Mass Action 2, etc... ugly but tolerable), but it would be beneficial
to get this supported.
