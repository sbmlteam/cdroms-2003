Martin 

Thanks for the explaination.

In my proposal I didn't have a module concept.
Instead I simply allowed a model to
have an optional set of ports which may or may not
be respected by a given model composition context.

I guess I would extend a submodel definition
to have an attribute (perhaps "modular" or "enforceInterface") which
allowed a model to state that its interface must be
respected.

This allows 3 types of submodels:

a) submodels without interfaces
b) submodels with interfaces that can be ignored
	completely or partially
c) modules with interfaces that must be respected.

> OK, definition of terms:
> 
> Submodel: separated part in an SBML model that contains an 
> own namespace for 
>           it's elements but does not define an interface.
>           For connections to the parent model or other 
> submodels you can
>           establish links to all inner parts of the submodel 
> by (possibly
>           hierarchical) model references.
> 
> Module:   also separated part (as the submodel). But defines 
> an accessible
>           interface. The parts in the interface can be safely linked
>           to other modules.
>           Everything else is hidden and can be changed by the modeler
>           without notice. This allows for different 
> implementations (detail)
>           or model improvements without breaking other 
> 'applications' of the 
>           module.
> 
>         Clear enougth?
