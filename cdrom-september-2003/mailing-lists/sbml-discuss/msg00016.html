On Thu, 5 Dec 2002, Andrew Finney wrote:

> On optional parameters
>
> I guess we're going down this route already as
> groups doing flux analysis don't require
> initialAmount on specie. Then its not a major step to
> make parameters optional.

Good.

> The only problem is that we're taking the model
> further away from what can be simulated which is
> unfortunate.  This makes establishing a regime of
> publishing models that simulate more problematic.
> I guess we can farily easily create an additionl
> stricter namespace and XML schema derived from the
> main standard which forces parameter values and
> initial values to be present.

We recognize that a model consists of two distinct parts: the "structure"
(equations, contraints, etc.) and the values to various parameters (that
term used loosely). Of course to do a simulation, there must be bindings
to all of the necessary values. But users need to be able to separate the
more "stable" structure from the more changable value settings.


> It may not be worth getting stressed by this,
> as lets face it, just having a parameter value
> (and thus a simulation) doesn't tell you anything
> about the correctness of the parameter value.
> A missing value is just a real bad value!

I vehmently disagree. Recognizing that a value is missing is FAR more
information to the reader than a bad value. If you know you have no value,
you can stick in a bad value if you want for simulation purposes, but at
least you know what you are doing so. If you are given a value, in the
current SBML you have no clue that it is a bad value (except hopefully in
the comments). As an analogy, in the numerical analysis community, NaN
(Not a Number) is a very important entity!

> A publishing regime would probably need to
> have a verification step where simulation
> results are compared to experimental data.

Yes, you might recall I proposed that eventually we need some sort of
definition language support for experimental data, among other things. Not
only the experimental data, but also objective functions for comparing
experimental data to simulation output should be expressable for a model.
Plenty for us to tackle later! :-)

> On the parameter sets concept.
>
> I can imagine sets of parameters being applied
> to models in much the same way as stylesheets
> to xml or html data.
>
> There are some issues with this:
>
> a) should we allow incomplete parameter sets?
> 	IMHO yes - it becomes a useful way to support
> 	models of mutations.  We may want to allow
> 	'stacks' of parameter sets to form a simulation
> 	input.

We do this extensively, so support for this would be welcome.

> b) does the model refer to the parameter set or visa-versa?
> 	certainly from a language development
> 	point of view it would be convient if
> 	the parameter sets refer to the model.
> 	That also appears to make sense from
> 	your requirements: a model has more than
> 	one parameter set and a parameter set is
> 	not meaningful without a model.  But a model
> 	could be considered useful without a parameter set.

Agreed.

> c) I'm not keen to bring parameter sets into the publishing
> 	senario at least not initially (you may disagree).  When a model
> 	is published IMHO the parameters should be transferred into a
> single
> 	model document.  (I guess this may break down in the mutation
> 	case.)  This will greatly simplify the curation process.

We certainly do use the concept of a suite of mutants, which generates an
ensemble of simulation runs.

> I think we could potentially move forward by creating a
> separate related standard for parameter sets that is almost
> entirely independent of the model standard.  Will this work?

Yes, so long as there is a recognized link between them. Such as one
referencing the other.

> On functions specifically for rate laws
>
> We should discuss this further with more people.
> IMHO this is an issue of presentation in your
> tool (but it may be common to many others)
> I suggest that, for the moment at least,
> you solve this by using a simple annotation.
> Do you need to communicate this specific information
> to other independantly coded tools?

That is probably what we will do. We can then see who else has an
interest.

> On functions with variable numbers of arguments
> (Mass Action)
>
> I think, probably needs further research,
> that you're right at the edge of what MathML
> supports right now.  If we want to fix this
> it will take some work putting this into
> the standard.  I say 'if' because a flexible
> scheme for this will introduce much complexity
> into the standard for perhaps little(?) pay off.
> I'm not unsymphaetic to your needs its just
> a question of where the big payoffs are.
> Feel free to make concrete suggestions of
> how the standard should be extended to tackle
> this issue.

I'm not positive... but I thought MathML supports variable-length
parameter lists for functions? If so, we would solve this problem just by
including that part of MathML in the standard. This is another case where
we can see who needs it. I am hopeful we are about to see more widespread
use of SBML, and that will test a lot of assumptions.

> Again you should examine if the use of
> annotations could reduce the uglyness
> of the current setup.

In this case, I think that is not how we will go. For now, I believe we
will simply code multiple versions of (for example) Mass Action.
