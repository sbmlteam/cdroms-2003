Folks

The current SBML Level 2 specification document doesn't
define the valid set of attributes and attribute values for
MathML elements.

I believe this is significant - its an oversight.

The following are suggested restrictions on the attributes
of MathML attributes.  The aim is to restrict semantics
to only those defined in SBML and MathML specifications;
and limit number types to those that will be actually used
in SBML models and SBML supporting simulators.

Please comment on this before
Wednesday 9am BST (UK Time).

Generic MathML attributes
-------------------------

attribute - definitionURL 
-------------------------
see
http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#id.4.3.2.3

URL or URI pointing to an alternative definition.

I suggest we ban this attribute on all elements apart from
csymbol.  This limits the semantics of MathML
element to those defined in the MathML and SBML specifications only
On csymbol this attribute can only take those values listed in the
SBML specification

attribute - encoding 
--------------------
see
http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#id.4.3.2.4

Syntax of the alternative definition.

I suggest we ban this attribute on all elements apart from
csymbol.  This limits the semantics of MathML
element to those defined in the MathML and SBML specifications only
On csymbol this attribute can only take the value "text"

attributes - class, style, id and other
---------------------------------------

All content elements support [these] general attributes that can be used
to modify the rendering of the markup

these are allowed by can be ignored by a parser

Attributes of cn elements
=========================

see http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#contm.cn

This is significant as it defines the forms/types
that can be used to define numbers in MathML expressions

attribute - type 
----------------
see
http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#contm.typeat
trib

Allowed values are "e-notation", "real", "integer", "rational",
"complex-cartesian", "complex-polar", "constant" 

The default value is "real"

I think we should restrict the valid types to "e-notation", "real",
"integer", "rational"

"constant" is effectively deprecated in the MathML V2 standard
we should use the constant elements instead.
The complex types don't seem useful in the current form of SBML L2.

To support "e-notation" and "rational" we need to introduce the 'sep'
element to the list of valid MathML element in SBML.
(rational is useful for supporting stoichiometry)

valid SBML examples:

<cn type="real"> 12345.7 </cn>
<cn> 12345.7 </cn>
<cn type="integer"> 12345 </cn>
<cn type="rational"> 12342 <sep/> 2342342 </cn>
<cn type="e-notation"> 12.3 <sep/> 5 </cn> 

note that the following is not valid in MathML:

<cn type="real"> 12345.7e4 </cn>

attribute - base 
----------------
see
http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#id.4.3.2.1

Number (CDATA for XML DTD) between 2 and 36. 

I suggest we ban this attribute - sticking with the default of 10

Attributes of ci elements
=========================

see http://www.w3.org/TR/2003/WD-MathML2-20030411/chapter4.html#contm.ci

in short I believe we should ban all attributes of ci elements:
Symbol type should always be inferred from the symbol's SBML definition.
