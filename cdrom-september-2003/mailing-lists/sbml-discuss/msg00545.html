Sven,

I was unable to find a schema for GraphXML( I looked at the Royere page, 
but the links were broken), although the GXL site I looked at: 
http://www.gupro.de/GXL/ seems remarkably well documented.  GXL seems 
like it is desgined to be more of a UML type language, since all of 
their examples show a UML translation. 

One thing none of these formats seem to address is the need to have one 
node with an associated set of data, but then be able to have that node 
in the graph multiple times, either as an alias like JDesigner, or in an 
embedded subgraph like me (Cytoscape).

In short I think that the first option presented below is still a good 
way to go.  But if someone has examples of how to get this functionality 
without nasty hacks from these existing languages, that would be swell.

--rowan

Sven Sahle wrote:

>Hello,
>
>I did only a quick check of the file formats Eric mentioned. GML is not XML. 
>The other are indeed XML dialects to describe graphs. While I did not check 
>if they can do everything that we might need, I am certain they can do lots 
>of stuff with graphs.
>This leads us to a decision:
>
>-Do we want to create a language to describe layouts for SBML models?
>In this case we need a language that emphasizes the biochemical structure of 
>the graph (while maintaining the flexibility to also represent additional 
>graphical objects). This is what our proposal aims at.
>
>-Or do we want to have a general graph description language with the 
>additional feature to somehow refer to elements of an SBML model. In this 
>case we could evaluate whether GXL or  GraphXML with extensions that refer to 
>SBML model objects is what we want.
>
>I do prefer the first. I think the layout language should know about concepts 
>like "graphical representation of a species" or "graphical representation of 
>a reaction".
>
