BPL is for bridging between C++ and Python, which are
different object oriented languages with *completely* different data structures.  
It's a challenge.  Although it can be simple for simple purposes, it is constructed with
dizzily advanced practices of template-based generic programming in C++, and so
it requests users to be C++ template experts.

> I designed libsbml from the beginning
> so that it could be wrapped by other languages.  In fact, my design
> and naming conventions were partly influenced by the recommendations
> of the Simplified Wrapper Interface Generator (SWIG; http://www.swig.org)
> project.  My hope is that SWIG will be viable for libsbml, as it supports
> many language targets:
> 
>        - Tcl
>        - Python
>        - Perl
>        - Java
>        - Ruby
>        - Mzscheme
>        - Guile
>        - PHP
> 
>      I know however that Boost is more advanced than SWIG in some areas,
> e.g., C-based shadow classes.  All of this will make for interesting
> discussions and investigations and I look forward to the wrapping
> expertise of your group.

libsbml is in C and seemingly it is designed so that language wrappers can be
 implemented in forms of straightforward function<->function mappings.    
SWIG is a good choice for this sort of a job.  I think BPL overkills in this case.
