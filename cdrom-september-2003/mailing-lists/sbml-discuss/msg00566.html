This are actually two alternatives we also considered. (ii) is of course what 
we would like to have for level 3. 
We didn´t chose suggestion (i) because we thought it is not terribly elegant 
to have an id in an annotation. But that does of course not mean that it can 
not be the best interim solution for level2. 

concerning the roles I somehow agree that it is not really graphical 
information. On the other hand it is not really clear that the information 
does not belong to the layout (see discussion that will propably follow Johns 
latest mail).

Independent from that there is still the problem I mentioned with A->B 
catalized by E displayed as A+E->B+E. I´m sorry I don´t remember who brought 
up that issue, but it was some weeks ago on the list. 
In our proposal an arbitrary number of graphical representations can refer to 
any model object. This is necessary because some (most?) layout programs can 
display a species as several nodes to avoid long curves and crossing lines. 
Same could apply to reactions (in a very complex reaction network you might 
want to display an important reaction several times). The same mechanism 
works for species reference glyphs and solves the above mentioned problem. 
The reaction A->B, enzyme E has 3 species references. If you want to render it 
like A+E->B+E the one modifier species reference is refered to by two species 
reference glyphs. One of these has role="substrate" and the other 
role="product". I think it makes sense to have this information in the 
species reference glyphs explicitly instead of only implicitly (in the 
definition of the curve).

By the way the reference to the model objects in the glyphs is optional in our 
proposal. While this might seem strange we think it makes sense. Say you have 
a very detailed sbml model. Then you want to draw a layout where several of 
the detailed reactions are displayed as one overall reaction. This reaction 
glyph would not refer to any single object in the model.  
Actually this means that the layout part of an sbml can be rendered without 
looking at the model itself.
