Hi all,

out of the discussions regarding clarifications of rule-semantics, 
I made the following notes. This can appear in the section
4.6.5 of the current SBML Spec.
One thing is still unclear: is it really intended to allow 
parameters being dynamic states (RuleType "rate")? 
The current spec makes this possible.

I would suggest to change the name `parameter' in this case 
(to quantity), since in most modelling applications parameters are 
usually constants or perhaps calculated intermediates but never 
dynamics.

* Assignmentrules are evaluated in order of appearance
* Assignmentrules must be ordered by the writing tool
  according to their dependencies and must not contain 
  _algebraic_ loops (like rule1: "a:= f(b,c)"
                          rule2: "c:= g(a)")
  Dependencies on dynamic (specie-|compartment-) values are allowed.
* AlgebraicRules allow to describe implicit algebraic constraints,
  but the whole system should stick to index 1 DAE (or is there 
  anyone working with automatic index reduction)
* If a SpecieConcentrationRule calculates a specie, this specie must 
  not have additional reactions (Otherwise we have at least index 2).

        Best regards
        Martin Ginkel
